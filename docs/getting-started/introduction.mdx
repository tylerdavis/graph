**graph** is a command-line workflow engine for AI-augmented operations, CI, automation and ETL. Workflows are defined by **plans**: small, composable, YAML documents that chain self-referential tool calls with typed dataflow between steps and LLM inference exactly where you place it. **graph** executes a plan identically every time: same steps, same order, on your laptop, on cron, or in CI.

```bash
# same steps, same order, every run — 0–1 LLM calls regardless of step count
graph plan run project_status '{"project":"New Relic to AWS Migration"}'
```

This is AI built for _operations_. An improvising agent gives you a different answer every run; a plan is an artifact — versioned in a repo, reviewed in a PR, validated before it runs, and cheap enough to schedule. That's the difference between a demo and a workflow you can run a thousand times.

## Why plans

- **Deterministic core, inference inline.** Steps are pure tool calls — zero LLM, however many there are. Inference enters exactly where you put it: [prompt tools](/tools/user-defined) as steps (classify, extract, summarize mid-plan) and a [finish](/plans/finish-modes) that is an LLM-written report, structured JSON for other tools, or nothing at all.
- **Repeatable and reviewable.** A plan is a file you check into a repo, review in a PR, and share with your team. No UI state, no prompt roulette — your Monday status report is the same report every Monday.
- **Composable.** Every plan joins the tool catalog as `plan__<identifier>`, invocable like any other tool — from the CLI, from scripts, or by name. And a plan's steps compose everything else in the catalog: MCP tools, wrapped CLIs, saved queries, prompt tools.
- **Generated, validated, tested.** You don't have to write plans by hand. `plan_and_execute` has an LLM planner author a plan on the fly — validated before execution, self-correcting against real errors when a step fails. Hand-written or generated, every plan is validated at load time (`graph plan validate`), and the [shape cache](/tools/shape-cache) teaches the planner what tools _actually_ return, so generated plans get better with use.
- **Bring your own models and storage.** Any MCP server; Anthropic, OpenAI, or OpenAI-compatible/local models, assignable per role (planner, solver, chat). A single binary with an embedded database — or pure in-memory storage for ephemeral CI runs. Your data stays on your machine.

## Built for scaled AI Ops

The same plan file works in a terminal, a cron job, and a CI pipeline: [finish modes](/plans/finish-modes) for reports, structured JSON, or silent side effects; meaningful exit codes; `${ENV}` secrets that fail loudly; repo-carried config (`./.graph/`); ephemeral storage. See [Automation & CI](/automation/ci).

## The chat surface

`graph ask` and `graph chat` run an agent loop over the same tool catalog. Treat it as the workbench: probe a tool's real output, prototype a workflow conversationally, seed the shape cache — then freeze what works into a plan. Details in [Chat & ask](/using/chat-and-ask).

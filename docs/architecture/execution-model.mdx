## The plan pipeline

Plans execute as: **validate ‚Üí run steps sequentially ‚Üí finish**.

- Step inputs render against prior results via the [template language](/plans/template-language) ‚Äî typed dataflow, zero inference.
- Four control steps are intercepted by the executor, never dispatched to a tool: [`exit`](/plans/escape-hatch) ends the plan early, [`decide`](/plans/branching) forks into one of two branches, and [`map`/`reduce`](/plans/iteration) run a body once per item of a list. All defer rendering: a `decide` renders only its gate up front and the non-taken branch never renders; `map`/`reduce` render only `over` up front and the body per item. Gate evaluations and body calls appear as ordinary flat tool events in traces. `map` is the pipeline's one concurrency point ‚Äî its `concurrency` knob runs independent items in parallel; everything else is sequential.
- The finish is a `solver` LLM call, an `output` render (no LLM), or nothing ([finish modes](/plans/finish-modes)).
- For `plan_and_execute`, a `planner`-role call authors the plan first, and defects loop back as replans (executed steps preserved). Human-authored plans skip planning and never replan.

## The agent loop

`ask` and `chat` run a tool-calling loop with the `chat`-role model:

```
message + tool catalog ‚Üí model
  ‚îú‚îÄ tool calls? ‚Üí execute (parallel within a round) ‚Üí results ‚Üí model again
  ‚îî‚îÄ text answer ‚Üí done
```

Bounded by `max_agent_iterations`. Tool failures return *into* the loop as error results ‚Äî the agent explains or works around them visibly.

## The anatomy of a chat turn using a plan

```
1. chat model    ‚Äî decides to call plan__project_status        (inference #1)
   ‚îú‚îÄ E0..E3     ‚Äî four Linear tool calls                      (no inference)
   ‚îî‚îÄ solver     ‚Äî writes the report from collected data       (inference #2)
2. chat model    ‚Äî reads the report, answers you               (inference #3)
```

The solver's report streams dim to stderr as progress; the agent's answer is the authoritative stdout. Yes, calls #2 and #3 partially overlap in purpose ‚Äî the agent layer adds condensing, conversation, and follow-ups. When you don't want that layer, `plan run` invokes the pipeline directly: one inference (solver) or zero (output/silent).

## Cost table

| Invocation | Inference calls |
|---|---|
| `plan run` (output / silent) | 0 |
| `plan run` (solver) | 1 |
| `exit` / `decide` gate | 0 (logical gate) or 1 judge call (`infer`) |
| `map` / `reduce` | 0 ‚Äî plus whatever the body calls |
| `ask` ‚Üí direct tools | 1 + one per tool round |
| `ask` ‚Üí plan tool | 3 |
| `ask` ‚Üí `plan_and_execute` | 4 + 1 per replan |

## Model roles

Every inference site resolves through `[models]` role assignment (`chat`, `planner`, `solver`, `repair` ‚Äî falling back to `default`), so cost tuning is pure config: strong model where judgment lives, fast model where volume lives. Structured outputs (planner, prompt tools) get one automatic `repair`-role fix-up attempt before erroring into the replan loop.

## Learning across turns

Every successful tool call feeds the [shape cache](/tools/shape-cache); the planner reads it fresh at each planning attempt. The system genuinely gets better at planning the more it's used ‚Äî including within a single run.

graph ships with — and defaults to — [LadybugDB](https://ladybugdb.com), an embedded graph database. Zero setup: the database appears at `~/.local/share/graph/db/` on first use.

## What's stored

| Data | Purpose |
|---|---|
| Threads & messages | conversation continuity (`--thread`), audit (`threads show`) |
| Tool shapes | the [shape cache](/tools/shape-cache) that trains planning |
| Entity graph | *(roadmap)* your synced domain data, queryable via Cypher |

`graph db query "<cypher>"` gives raw read access; [cypher user tools](/tools/user-defined) give curated access.

## Backends

Storage resolves through a backend abstraction, selected by `[storage].backend` or `GRAPH_STORAGE`:

| Backend | Character | Use when |
|---|---|---|
| `ladybug` (default) | persistent, embedded, **single-process** | daily interactive use |
| `memory` | ephemeral, lock-free | CI jobs, concurrent one-shots |

The single-process constraint is real: a running `graph chat` holds the database lock, and other invocations against the same `data_dir` fail with a clear error naming the holder. `memory` never touches the directory at all.

## Scaling path

The abstraction exists so centralized backends can slot in without re-architecture — the runtime depends only on a narrow storage trait. The planned trajectory:

- **Centralized state** (e.g. Postgres): threads and — most valuably — a **shared shape cache**, so every CI job inherits every other job's learned tool shapes.
- **Entity graph**: built-in `graph__cypher_query` / `graph__ingest` tools plus a `graph sync` flow that pipes MCP results into your own queryable graph; separable from state storage so a fleet can centralize one without the other.

Both are roadmap; the seams are already in place.

## Extensions

The engine's FTS and vector extensions ship **inside the graph binary**: the build fetches the matching extension binaries once (pinned to the engine version; set `GRAPH_LBUG_EXT_DIR` to a directory of pre-fetched `lib<ext>.lbug_extension` files for offline/hermetic builds), and at runtime they materialize under `<data_dir>/extensions/<version>/` and load by file path. No network at runtime, no dependency on the extension CDN being up — the runtime `INSTALL <ext>` download path (and its `~/.lbdb/` cache) is not used.

These back the future entity-graph features (full-text and semantic search over synced data); nothing loads them in the current CLI surface.

## Operational notes

- Back up by copying `data_dir` (it's just files).
- The [Ladybug Explorer](https://github.com/LadybugDB) container can browse the database — note it takes the same single-process lock while running.

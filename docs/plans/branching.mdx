A `decide` step forks the plan: it evaluates a gate ‚Äî the same gate grammar as an [exit gate](/plans/escape-hatch), spelled `if`/`infer` ‚Äî and runs the `then` branch when it holds, the `else` branch otherwise. Execution then rejoins the plan's normal step sequence. Where `exit` answers "should this plan stop?", `decide` answers "which action is correct next?" ‚Äî update or create, escalate or summarize, notify or stay quiet.

```yaml
steps:
  - id: E0
    tool_name: linear__list_issues
    input: { priority: 1, limit: 50 }

  - id: E1
    tool_name: decide
    input:
      if: { value: "{{E0.issues.length}}", op: gt, to: 5 }
      then:                                  # single tool call ‚Äî any tool, incl. plan__*
        tool_name: plan__escalate_triage
        input: { issues: "{{E0.issues}}" }
      else:                                  # inline step list, run in order
        - id: E10
          tool_name: user__summarize
          input: { text: "{{E0.issues}}" }
        - id: E11
          tool_name: user__notify
          input: { message: "{{E10.summary}}" }   # same-branch dataflow

  - id: E2                                   # rejoin ‚Äî same shape either way
    tool_name: user__log
    input: { entry: "took {{E1.branch}}: {{E1.result}}" }
```

## Gates

Exactly one of `if` or `infer` is required ‚Äî unlike `exit`, where omitting both means "unconditionally". The gate grammar is identical to [exit gates](/plans/escape-hatch) ‚Äî `decide` spells the logical gate `if` where `exit` spells it `when`, because `if ‚Ä¶ then ‚Ä¶ else` reads as the fork it is. `if` is a logical comparison (`eq ne gt lt gte lte empty not_empty contains`, with [typed splice](/plans/template-language#typed-splice) keeping numbers numeric); `infer` is a yes/no question judged by the `judge` model role. An `if` gate costs zero inference; an `infer` gate costs one cheap judgment call.

An inferred fork routes on judgment where no single comparison would do ‚Äî here, whether a status update reads as blocked:

```yaml
  - id: E2
    tool_name: decide
    input:
      infer: |
        Does this status update indicate the project is blocked?

        {{E1.statusUpdates.0.body}}
      then:
        tool_name: plan__escalate_blocker
        input: { project: "{{input.project}}", update: "{{E1.statusUpdates.0}}" }
      else:
        tool_name: user__log
        input: { entry: "on track" }
```

The judgment is a forced-schema call returning `{verdict, reason}`; `then` runs only on a yes, and the model's reasoning is carried in the step result ‚Äî `{{E2.reason}}` ‚Äî for downstream steps and the finish to use. Point the `judge` role at a cheap model; it falls back to `default`.

## Branches

`then` is required; `else` is optional. Each branch is either:

- **A single tool call** ‚Äî `{tool_name, input}`, no id. Any catalog tool works, including `plan__*` and `plan_and_execute`.
- **An inline step list** ‚Äî normal steps with ids. They run in authored order, and each may reference plan `input`, any earlier top-level step, and earlier steps *in the same branch*. Branch ids must not reuse top-level ids, and they are invisible outside the branch.

Branches may contain [`exit`](/plans/early-exit) steps ‚Äî a fired exit ends the **whole plan** from inside the branch (its `step` in the outcome carries the body path, e.g. `E4/then/bail`), and a passed gate lets the branch continue to its next step. Branches must not contain `decide`, [`map`, or `reduce`](/plans/iteration). For nested control flow, put it in a plan and call `plan__*` from the branch ‚Äî cycle detection and the depth cap apply as usual, and an error-exit inside that sub-plan surfaces as a failure of the decide step.

## The step result

The decide step's result ‚Äî stored under its own id ‚Äî is the only thing later steps see:

```json
{ "branch": "then", "verdict": true, "reason": null, "result": { ‚Ä¶ } }
```

- `branch` ‚Äî which side ran (`null` when the gate was false and there was no `else`).
- `verdict` ‚Äî the gate outcome; `reason` ‚Äî the judge's explanation, for `infer` gates.
- `result` ‚Äî the branch's output: the single call's result, or the **last** branch step's result.

Downstream references go through the decide id (`{{E1.result.‚Ä¶}}`, `{{E1.branch}}`), so the rejoined plan looks the same regardless of which branch ran. With no `else` and a false gate, `result` is `null` and the plan simply continues ‚Äî like a passed exit gate.

## Only the chosen branch is rendered

The executor renders the gate first, evaluates it, and only then renders the chosen branch ‚Äî per step, for inline lists. **The non-taken branch's templates are never evaluated.** That's what makes this pattern safe:

```yaml
input:
  if: { value: "{{E0.issues.length}}", op: gt, to: 0 }
  then:
    tool_name: linear__update_issue
    input: { id: "{{E0.issues.0.id}}" }     # only exists when then is taken
  else:
    tool_name: linear__create_issue
    input: { title: "{{input.title}}" }
```

If both branches rendered eagerly, `{{E0.issues.0.id}}` would raise `EmptyData` on an empty list ‚Äî the exact case `else` handles. Deferred rendering means each branch may assume it is the *right* branch. `EmptyData` raised inside the *chosen* branch still degrades normally ([errors](/plans/errors-and-replanning)).

## Failure and replanning

A failing branch call fails the decide step ‚Äî attributed as `step E1 (decide)` with the branch and inner tool named in the message. Human-authored plans fail hard as always; planner-authored plans replan with that context.

<Warning>
Branch step results are scoped to the branch and discarded after the decide step completes. When a planner-authored plan replans past a decide failure, a re-authored decide **re-runs the branch from its first step** ‚Äî including branch steps that succeeded before the failure. Keep branch steps idempotent (or route side-effecting sequences through a `plan__*` call guarded by its own exit gates) if duplicate execution would be harmful.
</Warning>

Branch steps count in `steps_executed` ‚Äî a decide with a two-step branch reports three executed steps (the decide plus two).

## Semantics summary

| Surface | Behavior |
|---|---|
| gate false, no `else` | result `{branch: null, verdict: false, result: null}`, plan continues |
| trace events | one `decide` tool event for the gate, then normal tool events for branch calls |
| `steps_executed` / envelopes | decide counts as 1, plus 1 per branch call or branch step run |
| branch tool fails | decide step fails: hard error (your plans) or replan (`plan_and_execute`) |
| cost | `if`: 0 inference; `infer`: 1 judge call ‚Äî plus whatever the branch calls |

`plan_and_execute`'s planner also has the decide tool, so LLM-authored plans can fork on their own intermediate results.

## The CI shape

A CI job wants: no interactive state, no disk residue, explicit inputs, machine-readable output, meaningful exit codes. That maps to:

```yaml
# GitHub Actions sketch
- name: Project status report
  env:
    ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
    LINEAR_API_KEY: ${{ secrets.LINEAR_API_KEY }}
    GRAPH_STORAGE: memory                 # ephemeral: no lock, no disk
  run: |
    graph plan run project_status @ci/inputs.json --json > status.json
```

## PR checks in one line

With the [github tool pack](/tools/user-defined#bundled-packs) enabled (`[tools] packs = ["github"]`), a plan can fetch its own PR context, gate on it, and post its own comment — so a merge check is a single step. `GRAPH_EVENTS=github` makes a failing plan annotate the run (`::error::` with the gate's message), and the [exit codes](/reference/environment#exit-codes) do the rest: 4 fails the check with the assertion, 1/3 fail it as a broken gate.

```yaml
env:
  GRAPH_STORAGE: memory
  GRAPH_EVENTS: github
  GH_TOKEN: ${{ github.token }}
  ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
steps:
  - uses: actions/checkout@v4
    with: { fetch-depth: 0 }        # plans diff refs locally
  # …install graph…
  - run: graph plan run docs_drift --input base="$BASE" --input head="$HEAD"
  - run: graph plan run pr_review --input pr="$PR"
```

This repo runs exactly this on its own PRs — see `.github/workflows/graph-checks.yaml` and the plans in `./.graph/plans/`.

## The pieces

**Ephemeral storage.** `GRAPH_STORAGE=memory` (or `[storage] backend = "memory"`) runs with no embedded database at all — nothing to lock, nothing to clean up. Threads don't persist and the [shape cache](/tools/shape-cache) starts cold per job; declared schemas and `tool_overrides` fill the gap for plans (which don't need shapes anyway — you authored the paths).

**Repo-carried config.** graph layers `./.graph/config.toml` over the global config, and loads `./.graph/plans/` + `./.graph/tools/` — so a repo can carry its entire graph setup:

```
repo/
  .graph/
    config.toml       # MCP servers, model roles (keys via ${ENV})
    plans/
      release_report.yaml
    tools/
      changelog.yaml
```

Secrets stay in `${ENV}` references — the config fails loudly if one is missing.

**Fixtures + overrides.** Check in the base inputs, override per run:

```bash
graph plan run release_report @ci/base.json --input version="$TAG"
```

**Branch on outcomes.**

```bash
graph plan run urgent_issues > urgent.json
count=$(jq '.count' urgent.json)
if [ "$count" -gt 0 ]; then …page someone…; fi

# exit 3 = inputs missing (caller bug), exit 1 = run failure, 0 = success
```

## Recipe: scheduled report to Slack

An output-mode or solver plan on cron, zero to one LLM calls:

```bash
graph plan run project_status '{"project":"New Relic"}' > report.md
curl -X POST "$SLACK_WEBHOOK" --data-urlencode "payload={\"text\": $(jq -Rs . < report.md)}"
```

## Recipe: silent action plan

A plan with [no solver and no output](/plans/finish-modes) performs side effects and exits — e.g. file a Linear issue for every failing canary from an exec tool's output. Zero inference, fully deterministic, reviewable in the PR that adds the plan file.

## Concurrency note

The default embedded database is single-process. CI jobs should use `memory`; on a shared box, give concurrent long-lived processes separate `data_dir`s. A centralized storage backend (shared shape cache across a CI fleet) is on the roadmap — the storage layer is already abstracted for it.

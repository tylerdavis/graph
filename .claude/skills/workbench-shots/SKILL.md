# Workbench docs screenshots

Every SVG in `docs/images/workbench/` is generated by executing a real headless
workbench session — real agent loop, pipeline, validator, and debugger; only the
LLM responses and tool I/O are scripted. Never hand-edit an SVG and never stage a
mockup: author a scenario, run the harness, and commit the scenario + SVG together.

The harness lives in `crates/graph-cli/src/workbench/shots.rs` (spec structs at the
top are the authoritative field reference). Scenarios live in `docs/shots/*.yaml`;
demo plans that exist only for screenshots live in `docs/shots/plans/`.

## Workflow

1. **Pick or write the plan.** Real repo plans (`.graph/plans/pr_review.yaml`) or a
   demo plan under `docs/shots/plans/`. Demo plans validate against the real
   catalog, so they may only use pack tools (`builtin__*`) and the repo's
   `.graph/tools/` user tools — **MCP tools (`linear__*` etc.) cannot be scripted**
   (no server is configured in the harness). Start a demo plan with a comment
   naming the scenario that uses it and "Not shipped to users."
2. **Write the scenario** (`docs/shots/<name>.yaml`) — spec below. Output lands at
   `docs/images/workbench/<name>.svg`.
3. **Generate:** `mise run shots` (equivalently
   `cargo test -p graph-cli shots -- --nocapture`). It regenerates every scenario;
   regeneration is byte-identical, so a diff in an SVG you didn't touch means
   rendering behavior changed — review that like code.
4. **Inspect the SVG as text** (don't guess from the browser alone) and iterate on
   `capture.keys` until the right row/pane is showing:

   ```bash
   python3 -c "
   import re
   t = open('docs/images/workbench/<name>.svg').read()
   els = re.findall(r'<text[^>]*x=\"([0-9.]+)\"[^>]*y=\"([0-9.]+)\"[^>]*>([^<]*)</text>', t)
   lines = {}
   for x,y,s in els:
       lines.setdefault(float(y), []).append((float(x), s))
   for y in sorted(lines):
       print(''.join(s for x,s in sorted(lines[y]))[:118])"
   ```

   (Spaces between spans are lost and the first border row renders garbled — both
   are artifacts of this reconstruction, not the SVG.)
5. **Embed** with a plain inline `<Frame>` + `<img>` — parameterized snippets don't
   interpolate props into JSX attributes, so there is deliberately no shared
   component. Give caption and alt real content:

   ```mdx
   <Frame caption="…what the reader should see…">
     <img src="/images/workbench/<name>.svg" alt="…same idea, self-contained…" />
   </Frame>
   ```

6. **Commit scenario + plan + SVG together.**

## Scenario spec

```yaml
name: my-shot            # output: docs/images/workbench/my-shot.svg
title: graph workbench — …   # window chrome title (ignored by crops)
plan: docs/shots/plans/demo.yaml   # omit → blank draft (drafting shots)
grid: { cols: 120, rows: 36 }      # default 110x32; 120x36 for full frames
draft_strategy: incremental        # optional; for drafting-overlay shots

chat: "…the user's message…"       # typed through the real reducer; omit → no agent turn

llm:                     # consumed IN ORDER across ALL roles (agent, solver, planner)
  - call: workbench__run_plan      # an agent tool call (optional text: before it)
    input: { input: { … } }        # run_plan also takes gated: true, breakpoints: [E1]
  - text: …                        # next LLM response: e.g. the solver's streamed report
  - text: …                        # then the agent's wrap-up
  - structured: { … }              # planner outline / step drafts (incremental drafting)

tools:                   # scripted outcomes per tool, consumed in call order
  builtin__git_changed_files:
    - { files: […], count: 2 }     # one entry per expected call — a map over N items
  builtin__git_file:               # needs N entries for its body tool
    - { text: "…" }

capture:
  at: finished           # or: loaded | pause_at: "E5" / "E1/do.1" | draft_step: 2
  keys: [tab, "1", j, j] # fed through the reducer after the capture state
  turn_seconds: 9.4      # optional status-bar timer
  crop: steps            # optional: chat | workspace | steps | body | status
```

Key tokens: any single character, plus `tab`, `enter`, `esc`, `up`, `down`,
`pgup`, `pgdn` (see `key_code` in shots.rs).

## Hard-won details

- Tool names and outcome shapes are validated against the real catalog — a typo'd
  tool or a result missing a schema-required field fails the harness with the
  problem named.
- **Plan-tab selection starts on the plan-root row**, so the first `j` selects the
  first step. Body rows and the `solver`/`output` row count as rows: for
  E0/E1/E2/E3(map)+body+solver, the solver row is six `j`s down.
- The detail pane shows, top to bottom: step name, input template, rendered input,
  result JSON — `pgdn` scrolls it. A solver row's result is the answer as an
  escaped JSON string; when you want the report as readable text, capture the run
  tab instead (`keys: [tab, "3"]`).
- `pause_at` answers earlier pauses with real `c` keypresses — a loop-line
  breakpoint steps iteration by iteration to the one you name (`E1/do.1`).
- Write scripted chat text that matches the final composition (don't have the
  agent say "see the detail pane" if the capture shows something else), and keep
  the fixture's story consistent with the surrounding docs page.
- `docs/shots/README.md` is the short human-facing version of this skill; update
  it if the workflow itself changes.
